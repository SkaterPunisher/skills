# tests-utils — формат тестов для утилит/чистых функций

## Канонические примеры

В первую очередь сверяйся с файлами в `./examples/` — это эталон для проекта, имеющий приоритет над абстрактными шаблонами ниже. Если стиль расходится — пиши как в примерах.

## Цель

Однотипные, читаемые unit-тесты на функции из `utils/helpers/lib`. Один файл с тестами на один файл с утилитами.
Каждая публичная функция покрывается отдельным `describe`. Внутри `describe` — `it` на каждый сценарий
(happy path + edge-cases).

## Стек

- jest 29
- ts-jest
- `@jest/globals` (`expect` импортируется явно)
- TypeScript 5

## Соглашения именования

- Файл: `<имя-файла>.test.ts` рядом с исходником (или в `__tests__/`).
- Импорт тестируемой функции — по абсолютному alias из проекта: `@shared/utils/...`, `@features/...` и т.п.
- `describe` — по-русски: `describe('проверка функции <имя>', ...)`.
- `it` — по-русски, в формате «возвращает X, если/когда Y» или «не делает Z, если Y».

## Обязательная шапка теста

```ts
import { expect } from '@jest/globals';
import { <имяФункции> } from '@<alias>/<путь>';
```

`expect` ВСЕГДА импортируется из `@jest/globals` (как в эталонных тестах).

## Скелет файла

```ts
import { expect } from '@jest/globals';
import { someFn } from '@shared/utils/functions/<group>';

describe('проверка функции someFn', () => {
  it('<что делает>, если <условие>', () => {
    const input = /* ... */;
    const result = someFn(input);
    expect(result).toBe(/* ... */);
  });

  it('возвращает <fallback>, если передан невалидный вход', () => {
    const result = someFn('abc' as never);
    expect(result).toBe('');
  });
});
```

## Что обязательно покрывать

Для каждой публичной функции — минимум:

1. Happy path (валидный вход → ожидаемый выход).
2. Все ветки `if/else`, видимые в реализации.
3. Невалидный вход: `undefined`, `null`, пустая строка, мусор (`'abc'`).
4. Граничные значения: для дат — некорректный месяц/день, лишние символы в годе, неверный разделитель.
5. Если функция возвращает `null`/`''`/`[null, null]` как fallback — отдельный `it` на КАЖДЫЙ сценарий fallback.

## Структура `expect` в зависимости от возврата

- Примитивы → `toBe`.
- Объекты/массивы → `toEqual`.
- `Date` → `toEqual(new Date(year, monthIndex, day))`. Помни про monthIndex: июль = 6.
- `null` → `toBeNull()`.
- Booleans → `toBe(true)` / `toBe(false)`.

## Антипаттерны (не делать)

- НЕ писать один гигантский `it` со всеми кейсами — каждый кейс отдельным `it`.
- НЕ использовать `it.each` без необходимости — у нас явные `it` с осмысленными названиями (формат проекта).
- НЕ мокать ничего внутри тестов на чистые функции. Если функция дёргает сеть/таймеры — она не утилита,
  и тестируется по другому скиллу.
- НЕ импортировать тестируемую функцию относительным путём, если в проекте есть alias.

## Чек-лист перед сдачей

- [ ] Один `describe` на одну функцию. Заголовок — `проверка функции <имя>`.
- [ ] `expect` импортирован из `@jest/globals`.
- [ ] Покрыт happy path + все edge-cases из реализации.
- [ ] Названия `it` читаются как «что/когда». На русском.
- [ ] Нет моков, нет сети, нет таймеров.
- [ ] `toEqual` для объектов/массивов/`Date`, `toBe` для примитивов.
- [ ] Команды локально: `npm run test:only-changed`, `npm run test:watch`.
