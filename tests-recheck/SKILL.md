# tests-recheck — воркфлоу «проверь и почини тесты у <name>»

## Канонические примеры

При наличии файлов в `./examples/` (например, `transcript-success.md`, `transcript-ambiguous.md`) — итоговый отчёт пользователю должен повторять их форму. Если папка содержит только `PLACEHOLDER.md` — следуй описанию ниже.

## Команда прогона

В этом проекте конкретный тест-файл запускается так:

```bash
npx jest --testPathPattern=<name>
```

`<name>` — то имя, которое назвал пользователь. Это ИЛИ имя функции/компонента, ИЛИ часть пути.
Jest подхватит любой файл, чей путь содержит подстроку.

Полезные дополнительные флаги:

```bash
npx jest --testPathPattern=<name> --watch                  # пересборка при сохранении
npx jest --testPathPattern=<name> -t "<имя it/describe>"   # конкретный it/describe
npx jest --testPathPattern=<name> --coverage=false         # быстрее, без coverage
npx jest --testPathPattern=<name> --runInBand              # последовательный прогон, читаемые логи
```

Для общего прогона см. `package.json`: `npm run test`, `npm run test:only-changed`, `npm run test:watch`.

## Алгоритм

### 1) Найти файлы

По имени `<name>` найти и **исходник**, и **тест**:

- Тест-файл: ищи `**/<name>.test.ts` и `**/<name>.test.tsx`. Если не нашлось, ищи по слабому совпадению:
  `find . -type f \( -name "*<name>*.test.ts" -o -name "*<name>*.test.tsx" \)`.
- Исходник: рядом с тестом, или ищи `**/<name>.ts(x)`. У утилит может лежать в `index.ts` папки —
  тогда смотри экспорты.

Если тест не найден — это не «починка», а «написание тестов с нуля». Сообщить пользователю и
переключиться на соответствующий скилл (`tests-utils`/`tests-components`/...).

### 2) Определить тип теста

По исходнику определи, какой формат должен использоваться, и подгрузи соответствующий скилл:

| Что в исходнике | Скилл-формат |
|---|---|
| Pure-функция (никакого React/JSX/сети) | `tests-utils` |
| `import { create } from 'zustand'` | `tests-store` |
| Хук `useQuery`/`useMutation`/`useSWR` | `tests-queries` |
| Кастомный хук без сети/стора | `tests-hooks-generic` |
| Схема yup/zod | `tests-validation-schemas` |
| Простой UI-компонент (без моков) | `tests-components` |
| Контейнер с моками хуков и обёрток | `tests-components-mocked` |
| Целиком форма Formik (есть submit-кнопка) | `tests-formik-form` |
| Использует `react-router-dom` | `tests-routes` (часто комбинируется) |

### 3) Запустить тест и собрать падения

```bash
npx jest --testPathPattern=<name> --runInBand
```

Прочитать вывод. Зафиксировать список упавших `it` и **причины**:
- `Expected X, received Y` → значение не совпадает.
- `Cannot find module 'X'` → импорт сломался (alias/переименование).
- `TypeError: ... is not a function` → API изменился.
- `An update to <Component> inside a test was not wrapped in act` → не хватает `await` или `act` в правильном месте.
- `Timeout - Async callback was not invoked` → не отдан Promise/нет `await waitFor`.

### 4) Решить: чинить **тест** или **исходник**

Это критично. Определи это **до** правки:

- Если **бизнес-логика поменялась** (пользователь сам только что изменил исходник) → почти всегда чинится **тест**.
  Узнаваемый признак: исходник недавно правился, тест стоит со старыми константами/полями.
- Если **тест уронил неработающий код** (баг в исходнике, тест корректен) → чинится **исходник**.
  Узнаваемый признак: assert описывает разумное поведение, но реальный возврат другой.
- Если **API сломался у обоих** (переименование) → чинятся оба.

Если непонятно — спроси пользователя одной короткой фразой: «Тест ловит реальный баг, или
поведение изменилось намеренно? Поправлять тест или исходник?». **Никогда** не угадывай молча, это
потом тяжело откатить.

### 5) Починить

При правке **теста** строго следуй формату из соответствующего скилла из таблицы выше:
- `expect` импортируется из `@jest/globals`;
- `describe`/`it` на русском;
- `userEvent.setup()` для UI-тестов;
- свежий `QueryClient` для react-query;
- `jest.clearAllMocks()` в `beforeEach`;
- и т.д.

Не «оптимизируй» соседние тесты, которые не падают — только минимально необходимая правка.

### 6) Прогнать снова

```bash
npx jest --testPathPattern=<name> --runInBand
```

Если зелёное — отчитаться:
- какие `it` упали и почему;
- что было поправлено (тест/исходник);
- ссылки/имена файлов.

Если красное — повторить с шага 3, не более **3 циклов**. После 3-х неудач — остановиться и спросить
пользователя, что делать дальше.

### 7) Финальная сверка (если правились несколько файлов)

Прогнать связанные тесты целиком, на случай если правка задела соседей:

```bash
npm run test:only-changed
```

(или `npm run test`, если правок много).

## Что обязательно показать пользователю

После работы — короткий итог:

1. **Что нашёл**: путь к тесту и исходнику.
2. **Что упало и почему** (1-2 строки на каждое падение).
3. **Что поправил**: тест или исходник, какие именно `it` / какие строки.
4. **Команда воспроизведения**:
   ```bash
   npx jest --testPathPattern=<name>
   ```
5. **Финальный статус**: зелёное / остаются падения (тогда какие).

## Антипаттерны (не делать)

- НЕ удалять упавший `it` «чтобы прошло». Если он невалиден — заменить на корректный, не выкидывать.
- НЕ менять снапшоты руками без ясного понимания, почему изменился вывод.
- НЕ править исходник, чтобы тест прошёл, если тест ловит реальный баг. Сначала свериться с пользователем.
- НЕ комментировать упавшие `it` — это вранье в зелёном CI.
- НЕ запускать весь `npm run test` ради одного файла на этапе починки — это медленно. Только в финальной сверке.
- НЕ забывать про прогон с `--runInBand` для читаемого вывода ошибок (без него jest пишет логи параллельно
  и тяжело понять, что упало).

## Чек-лист перед сдачей

- [ ] Прогнал `npx jest --testPathPattern=<name>` — стало зелёно.
- [ ] Если правил исходник — это согласовано с пользователем.
- [ ] Формат теста соответствует нужному `tests-*` скиллу.
- [ ] Не закомментирован/удалён ни один упавший `it` без замены на корректный.
- [ ] Финальная сверка `npm run test:only-changed` (если правок несколько) — зелёная.
- [ ] Пользователю отдан короткий отчёт: что нашёл, что упало, что поправил, как воспроизвести.
