# Канонические примеры — `tests-recheck`

Этот скилл — воркфлоу, а не формат. Эталонных тестов у него нет. Но один пример очень помогает: «как должен выглядеть отчёт после прогона».

## Что положить

### Желательная пара

- `transcript-success.md` — пример хорошего короткого отчёта после успешной починки. Что было найдено, что упало, что поправлено, какая команда воспроизводит результат.
- `transcript-ambiguous.md` — пример отчёта, когда падение неоднозначно: скилл задал пользователю вопрос «чинить тест или исходник?» и дождался ответа.

### Шаблон `transcript-success.md`

```
## Что нашёл

- Тест: src/features/payments/utils/__tests__/mappedTransfersToTableData.test.ts
- Исходник: src/features/payments/utils/mappedTransfersToTableData.ts

## Что упало и почему

- `it('преобразует transfer в строку таблицы')` — упало на assert: `expect(result.amount).toBe('100.00')`,
  фактический результат `'100'`. Причина: после рефакторинга `formatAmount` перестал добавлять `.00` для целых чисел.

## Что поправил

- Поправлен **тест** (поведение изменилось намеренно): `expect(result.amount).toBe('100')`.

## Команда воспроизведения

\`\`\`bash
npx jest --testPathPattern=mappedTransfersToTableData
\`\`\`

## Финальный статус

Зелёное. 4/4 it прошло.
```

### Шаблон `transcript-ambiguous.md`

Покажи структуру, когда скилл НЕ принял решение сам, а спросил пользователя. Это нужно, чтобы qwen не «уверенно угадывал», а явно поднимал развилку.

## Как сослаться из SKILL.md

В `../SKILL.md` уже добавлен раздел `Канонические примеры` со ссылкой на эту папку. Когда положишь — отчёты qwen начнут совпадать по форме с этими шаблонами.

## Что не нужно

- Не клади примеры самих тестов — для них есть отдельные `tests-*` скиллы со своими `examples/`.
- Не клади полные логи jest — только осмысленный человекочитаемый отчёт.
